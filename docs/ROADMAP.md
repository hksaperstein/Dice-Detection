# Roadmap / Open Items

## 2026-07-07 — user feedback batch (not yet investigated/fixed)

Direct user feedback, verbatim intent preserved. None of these are fixed yet — this is the entry point for next session.

1. **"Validation techniques are weak" — several assets are missing indices (numerals) entirely.**
   This is a real, distinct gap from what's currently tracked. `mesh_quality_warnings` (non-manifold/degenerate checks) and `engraving_warnings` (EXACT+FLOAT both collapsed/debris) do NOT catch the case where a boolean cut "succeeds" by those checks (no collapse, no debris) but removes a negligible/invisible amount of material — this exact failure mode is already an acknowledged, unresolved gap documented in `_boolean_diff_apply`'s own docstring (the "Δ"/"Ζ" near-zero-effect case from earlier this project), and a prior attempt to catch it via cutter-volume-ratio was found too noisy to ship (67% false-trigger rate). **Next step:** design a real "is a numeral actually present/visible" check — likely a geometric check (search for recessed vertices near each cut's planned position/depth from Phase 1, not just topology validity) rather than another volume-ratio heuristic. Consider a fresh-reload render+pixel check per engraved face (mirroring the decal path's existing pixel-verification technique) as an alternative or supplement.

2. **Pink overused in generated materials.** Likely a sampling-distribution issue in `sampler.py`'s HSV ranges (`_sample_material_params` or similar) — needs a real data check (tabulate hue distribution across a batch) before assuming/guessing a fix, per this project's own "verify before fixing" convention.

3. **Engraved dice should use 2-tone colors/patterns**, not just a flat contrasting fill. Current mechanism: `_assign_fill_material_to_recessed_faces` + `build_fill_material` gives recessed faces a second, distinct material slot. Investigate whether this needs to be MORE visually distinct (2-tone/patterned) than it currently is, or whether this is asking for a new capability (patterned fill materials, not just a flat second color).

4. **Add additional fonts.** Currently only 3 (`FONT_FILES` in `glyphs.py`: LiberationSans-Bold, LiberationSerif-Regular, LiberationSansNarrow-Bold). Survey `/usr/share/fonts/` for more real, distinct, legible options across styles (serif/sans/condensed/display) before arbitrarily picking more.

5. **`GREEK_NUMERALS` dict (`glyphs.py:90-96`) is confirmed broken/nonsensical.** Found via direct code inspection this session: `0: "Ω"` (Omega) is used for zero, which is NOT a real Greek numeral convention — the ancient Greek Milesian/Ionic numeral system this dict is loosely modeling has no zero. Values 10-18 use an invented "Ι + unit-letter" compound (`ΙΑ`, `ΙΒ`, ... `ΙΘ`) that resembles but does not exactly match real Milesian numeral conventions, and `15: "ΙΣ"` uses Sigma where the real system uses digamma/stigma (Ϛ) for 6, not a unit-15 compound with Σ — this specific entry looks like a plain error, not a stylization choice. **Next step:** delegate real research (delegating-technical-research skill) on the actual historical Greek (Milesian/Attic acrophonic) numeral system before rewriting this dict, rather than inventing another approximation. Consider whether Attic acrophonic (simpler, uses Η/Δ/Η/Χ/Μ for 100/1000/etc., closer to what a d20 needs) is a better real-world fit than Milesian.

6. **Engravings should be "way way shallower and softer."** `ENGRAVE_DEPTH_FRACTION` is currently `0.02` (already reduced twice this project: 0.04→0.03→0.025(reverted)→0.03→0.02). A further reduction needs testing against `_boolean_diff_apply`'s existing collapse/debris checks — shallower cuts are more prone to the EXACT solver's near-zero-effect no-op failure mode (see item 1), so this and item 1's validation fix are linked: can't safely go shallower without a real "did the numeral actually engrave" check in place first.

   "Softer edges": the previous approach (bevel the CUTTER mesh before the boolean cut) was tried and reverted this session — it inverted the boolean operation (added material instead of subtracting) for any digit with an enclosed hole (6, 8, 0), across 6 different configurations tried. **Do not retry that approach as-is.** A more promising untried direction: bevel the RECESS BOUNDARY on the DIE, after the cut (not the cutter, before) — mirroring the technique that fixed the structural-edge bevel bug (mark edges with a `bevel_weight_edge`-style attribute, but this time mark the newly-created recess boundary edges post-cut, not the pristine polyhedron's edges pre-cut), then run a small WEIGHT-based bevel pass scoped to just those edges. Needs Blender Python research on the most robust way to identify "new edges introduced by this specific boolean cut" (a topology diff between pre-cut and post-cut bmesh state is one option) before attempting.

7. **"You can probably do more Blender Python research and learning."** General feedback to invest more in researching established Blender/bpy techniques (via delegating-technical-research) before ad-hoc trial-and-error, especially for: text-on-mesh/boolean-cut best practices, robust normal-consistency handling for multi-boundary meshes (see the recalc_face_normals recurrence noted in memory), and material/shader authoring patterns for procedural 2-tone/patterned fills.
